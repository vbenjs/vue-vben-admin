# 客户个性化需求能力设计

- 日期: 2026-04-06
- 状态: 已确认设计，待进入实现计划
- 适用仓库: `RISS`
- 目标范围: 多租户标准产品中的客户级页面个性化、轻量业务策略个性化、个人偏好个性化

## 1. 背景与目标

当前仓库已经具备以下基础能力:

- 租户主数据和登录上下文, 包含 `tenantId`、`tenantName`、`fiscalYear`
- 页面运行态分层能力, 包含标准模板、租户覆盖、个人偏好
- 页面模板初始化、预览、发布、回滚和日志能力

现阶段需要把“不同客户的个性化需求”从零散定制提升为标准产品能力, 目标是:

- 让同一套标准产品支持不同客户的页面结构差异
- 让不同客户具备受控的字段行为和业务参数差异
- 保持标准产品可升级、可发布、可回滚、可测试
- 避免项目演变为按客户分叉代码的交付模式

## 2. 设计原则

- 分层: 标准模板、租户覆盖、租户策略、个人偏好职责清晰
- 受控: 不允许租户直接注入脚本或任意业务代码
- 一致: 前后端共享相同配置语义, 以后端校验为准
- 可运营: 所有配置均支持草稿、预览、发布、回滚、日志
- 可扩展: 第一阶段先覆盖高频需求, 为后续复杂规则预留扩展点

## 3. 第一阶段能力边界

### 3.1 纳入范围

第一阶段支持四类能力:

1. 页面结构个性化
   - 查询项显隐、顺序、宽度、文案
   - 表格列显隐、顺序、宽度、标题
   - 表单分组显隐、顺序、标题
   - 工具栏按钮显隐、顺序、名称
   - 详情区块和弹窗模式配置
2. 字段行为个性化
   - 必填
   - 只读
   - 默认值
   - 占位提示
   - 候选范围限制
   - 简单联动显隐
3. 租户业务参数个性化
   - 默认年度
   - 默认币种
   - 预算控制模式
   - 默认付款方式
   - 辅助核算维度
   - 打印模板绑定
   - 附件规则
4. 个人偏好个性化
   - 列宽
   - 列顺序
   - 默认筛选条件
   - 默认视图

### 3.2 不纳入范围

第一阶段明确不做以下能力:

- 租户自定义数据库字段
- 租户自定义审批状态机
- 租户执行任意脚本或表达式
- 租户独立业务分叉
- 通用低代码规则引擎

## 4. 推荐方案

本次设计采用“页面方案扩展型 + 轻量租户策略层”的组合方案。

### 4.1 方案说明

- 保留现有页面运行态能力: `sys_page_template`、`sys_page_override`、`sys_user_page_preference`
- 在页面方案之外新增“租户策略层”, 用于承载字段行为、动作权限、附件要求、打印模板绑定等轻量规则
- 页面负责描述“长什么样”
- 策略负责描述“允许做什么、必须填什么”
- 个人偏好只负责“用户习惯”, 不允许突破租户策略约束

### 4.2 选择理由

- 贴合当前仓库已有能力, 成本最低
- 能覆盖大部分客户差异化诉求
- 升级风险可控, 不需要按客户分叉代码
- 便于先在财务试点页验证, 再向合同、科研、采购扩展

## 5. 总体架构

### 5.1 分层模型

运行态按以下层次组合:

1. 平台标准模板
2. 租户页面覆盖
3. 租户策略配置
4. 个人偏好

其中:

- 模板、租户覆盖、个人偏好共同产出最终 `schema`
- 租户策略产出最终 `policy`
- 个人偏好仅能影响便利性配置, 不能覆盖租户策略的强约束

### 5.2 统一运行态返回

后端统一向前端返回:

```json
{
  "pageCode": "finance.reimbursement.query",
  "pageName": "报销单查询",
  "context": {
    "tenantId": 1,
    "tenantName": "默认账套",
    "fiscalYear": "2026"
  },
  "schema": {},
  "policy": {},
  "sources": {},
  "versions": {}
}
```

建议字段定义:

- `schema`: 页面结构最终结果
- `policy`: 字段行为、动作权限、附件规则、打印模板等最终结果
- `context`: 租户、用户、年度上下文
- `sources`: 模板、覆盖、策略、偏好来源标识
- `versions`: 模板、覆盖、策略、偏好当前版本

### 6.3 策略 JSON 结构

第一阶段只支持声明式规则:

```json
{
  "fields": {
    "form.basic.amount": {
      "required": true,
      "readonly": false
    },
    "form.basic.projectType": {
      "defaultValue": "财政项目"
    }
  },
  "actions": {
    "toolbar.export": {
      "visible": false,
      "enabled": false
    }
  },
  "attachments": {
    "attachment.invoice": {
      "required": true,
      "maxCount": 10
    }
  },
  "print": {
    "defaultTemplate": "reimbursement-a4-v2"
  }
}
```

第一阶段不支持脚本和表达式规则, 只允许:

- `visible`
- `enabled`
- `required`
- `readonly`
- `defaultValue`
- `allowedValues`
- `placeholder`
- `maxCount`
- `templateCode`

## 7. 运行时合并与校验

### 7.1 合并顺序

建议统一运行顺序:

1. 读取页面标准模板
2. 合并租户页面覆盖
3. 合并租户策略
4. 合并个人偏好

限制规则:

- 个人偏好可以改变列宽、顺序、默认筛选
- 个人偏好不能把租户禁用字段改成可编辑
- 个人偏好不能打开租户已隐藏的强约束动作

### 7.2 前后端职责

前端负责:

- 结构渲染
- 交互体验
- 字段显隐和默认值展示
- 按钮和入口按策略显示

后端负责:

- 必填校验
- 只读字段保护
- 枚举值和候选范围合法性
- 动作权限判定
- 附件规则校验
- 打印模板合法性校验
- 租户和年度隔离

安全原则:

- 前端限制只提升体验
- 所有强约束均以后端校验结果为准

### 7.3 错误返回

策略相关错误建议统一返回:

- `code`
- `message`
- `policyKey`
- `policyVersion`

用于前端准确提示“当前客户策略不允许该操作”, 而不是泛化成系统异常。

## 8. 生命周期设计

页面模板、租户覆盖、租户策略统一采用相同生命周期:

1. 草稿
2. 预览
3. 发布
4. 回滚

要求:

- 发布前可预览
- 发布后记录版本快照
- 支持版本回滚
- 可追踪操作人、时间、备注

## 9. 仓库落地方案

### 9.1 后端改造点

- 保持 `apps/server/src/sys-page-schema` 作为页面模板、覆盖、偏好和运行态中心
- 保持 `apps/server/src/sys-tenant` 作为租户主数据中心
- 复用 `apps/server/src/auth` 中的登录上下文
- 新增 `apps/server/src/sys-tenant-policy`
- 在试点业务模块中接入策略兜底校验

后端实施建议:

1. 先新增表和策略模块接口
2. 再扩展统一运行态接口, 补充 `policy`、`context`、`sources`、`versions`
3. 最后逐步把策略校验接入业务接口

### 9.2 前端改造点

- 保留并升级 `apps/web-antd/src/composables/usePageSchema.ts`
- 建议新增统一运行态消费入口, 如 `useRuntimePageConfig`
- 基于 `apps/web-antd/src/views/sys/form-design/page-meta.vue` 扩展个性化控制台
- 保留 `apps/web-antd/src/views/sys/tenant-parameter/index.vue` 管理标量参数
- 新增租户策略管理页, 不把结构化策略硬塞进租户参数页

前端页面接入目标:

- 页面只声明稳定 `key`
- composable 返回已解析配置
- 页面组件消费最终结果, 不自行解释底层层次关系

## 11. TDD 实施策略

正式实现时必须先测后写, 推荐顺序如下:

1. 后端单测
   - 运行态合并顺序
   - 租户策略合并
   - 个人偏好不得突破租户策略
   - 发布、回滚、版本记录
   - 空策略、非法策略、禁用策略边界
2. 前端 composable 测试
   - `schema` 和 `policy` 解析
   - 字段显隐和只读
   - 按钮可用性
   - 默认值和默认筛选回填
3. 试点页面接入
   - 先改一个页面跑通链路
   - 再扩展到第二个和第三个页面

实施顺序要求:

- 先做解释器和运行态接口
- 再做试点页消费
- 最后做策略管理界面增强

不建议先做“大而全”的后台配置页, 否则容易出现“可配置但不可消费”的失衡状态。

## 12. 风险与控制措施

### 12.1 主要风险

- 页面 `key` 不稳定, 导致覆盖和策略失效
- 前端和后端对同一策略语义理解不一致
- 个人偏好覆盖了租户强约束
- 配置能力过快膨胀, 演变为低代码规则引擎
- 一次性覆盖过多页面, 回归成本过高

### 12.2 控制措施

- 为所有可配置元素建立稳定 `key` 规范
- 统一运行态接口, 避免前端自行拼装配置
- 在后端实现最终校验兜底
- 第一阶段仅开放声明式规则
- 按试点节奏分波推进

## 13. 验收标准

满足以下条件时, 认为第一阶段设计落地成功:

- 同一页面在不同租户下能呈现不同结构和字段行为
- 页面差异不需要改业务代码即可生效
- 关键业务约束能被后端策略兜底校验
- 模板、租户覆盖、策略都支持预览、发布、回滚、日志
- 试点页面具备自动化测试覆盖
- 后续扩展到合同、科研、采购时复用模式清晰
